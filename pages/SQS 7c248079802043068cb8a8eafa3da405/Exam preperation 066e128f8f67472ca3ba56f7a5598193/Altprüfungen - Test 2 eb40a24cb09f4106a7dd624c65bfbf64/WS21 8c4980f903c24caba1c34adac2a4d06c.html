Mock vs. Stub

Ein System nützt die folgende Schnittstelle zum Suchen nach Events:

interface OnlineEventService {/**
* Searches for events in the given location on the given date.
*
* @param location must not be null
* @param date must not be in the past
* @return a list with the results
*/
List<String> find(String location, LocalDate date);
}

Um das System isoliert mit Unit-Tests zu testen wurde folgender Test Double implementiert:

public class SimpleEventService implements OnlineEventService {
private List<String> events;
private boolean called;public void setEvents(List<String> events) { this.events = events; }
public boolean isCalled(){ return called; }@Override
public List<String> find(String location, LocalDate date) {
Assertions.assertNotNull(location);
Assertlons.assertNotNull(date);
Assertions.assertTrue(date.isAfter(LocalDate.now());
called = true;
return events;
}
}

Handelt es sich dabei um ein Stub oder ein Mock?

Begründen Sie Ihre Antwort und erläutern Sie den Unterschied basierend auf dem gegebenen Beispiel.

Es handelt sich um ein Mock, da die Argumente die an das Mock gegeben werden auch überprüft werden.

Ein Mock erkennt wenn unerwartete Werte erhalten werden - mit Assertions und Exceptions.

Fehler:

  1. Keine Testisolation

    Die Datenstrukturlist wird innerhalb aller Aufrufe benützt.

  1. Kein Testen

    Die erste Test-Methode prüft die Variable isFoundnicht mit asserts ab .

Stringbasierter Stack, JUnit

Bei der Klasse StringStack handelt es sich um eine stringbasierte Stack Implementierung.

public class StringStack
private String[] storage;
private int position;public StringStack(int capacity) { this.storage = new String[capacity]; }public int size() { return position; }// doesn't check whether position < storage.length()
// can cause IndexOutOfBound
public void push (String elem) {
storage[position] = elem;
position++;
}public String pop() throws EmptyStackException {
if (position == 0) { throw new EmptyStackEx ception (); }String elem = storage[position-1];
storage[position-1] = null;
position--;
return elem;
}
}

3 wichtige nichtfunktionale Anforderungen / Qualitätsanforderungen

  1. Usability

    Das System sollte gut dokumentiert sein, sodass sie weiterskalieren kann.

    Das System sollte von weniger Technik-affinen Nutzenden wie Schülern leicht nutzbar sein um ihre Stundenpläne nachzuschlagen. Das muss mit qualitativen Untersuchungen festgestellt werden.

  1. Reliability

    Bei einem Systemausfall sollten Administratoren benachrichtigt werden und es muss die Möglichkeit bestehen die Daten zur Anwesenheit / Abwesenheit auch lokal im System zu speichern und nachträglich hochzuladen wenn das System wieder erreichbar ist.

  1. Supportability

    Es wurde bereits beschlossen, dass der Funktionsumfang erweitert werden sollte.

    Deshalb sollten statische Code Analysewerkzeuge die den Code nach coding-conventions untersuchen wie SonarLint mit den Conventions von Google eingesetzt werden.

    Dadurch wird gewährleistet, dass das Projekt weiterskalieren kann.
